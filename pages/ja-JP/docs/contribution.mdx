# fastexcel への貢献

fastexcel はコミュニティのすべてのユーザーと開発者を歓迎します。問題の報告、ドキュメントの改善、コードの提出、技術サポートの提供など、あなたの参加は fastexcel をより良くするのに役立ちます。

---

## 問題の報告

fastexcel を使用する際にフィードバックを提供することをお勧めします。問題を報告するには、[NEW ISSUE](https://github.com/CodePhiliaX/fastexcel/issues/new/choose) を使用して問題を提出できます。

### 高品質な問題報告

コミュニケーション効率を向上させるために、問題を提出する前に次のことをお勧めします：  
1. **既存の問題を検索する**：問題が既に報告されているかどうかを確認します。存在する場合は、新しい問題を作成するのではなく、既存の問題に詳細情報をコメントで追加してください。  
2. **問題テンプレートの使用**：問題テンプレートは [ISSUE TEMPLATE](./.github/ISSUE_TEMPLATE) にあります。テンプレートの要件に従って記入して、問題の説明が正確かつ完全であることを確認してください。

以下の場合に新しい問題を提出することが適しています：
- バグの報告
- 新機能の要求
- パフォーマンスの問題
- 機能提案または設計
- ドキュメントの改善
- テストカバレッジの最適化
- 技術サポートが必要
- プロジェクトに関連するその他の問題

> **注意**：パスワード、キー、サーバーアドレス、個人データなどの機密情報を問題に含めないでください。

## コードとドキュメントの貢献

fastexcel へのすべての改善は Pull Request（PR）を介して行うことができます。バグ修正、コードの最適化、機能の強化、ドキュメントの改善など、どんな貢献も大歓迎です！

### 貢献できる領域
- 誤字の修正
- バグ修正
- 冗長なコードの削除
- テストケースの追加
- 機能の強化
- コードの可読性向上のためのコメントの追加
- コード構造の最適化
- ドキュメントの改善または補完

**原則**：**プロジェクトの改善に役立つすべての PR は奨励されます！**

PR を提出する前に、次のガイドラインに精通してください：
1. [ワークスペースの準備](#ワークスペースの準備)
2. [ブランチの定義](#ブランチの定義)
3. [コミットルール](#コミットルール)
4. [PR の説明](#prの説明)

---

### ワークスペースの準備

GitHub アカウントを登録し、次の手順に従ってローカル開発環境を設定していることを確認してください：
1. **リポジトリをフォーク**：fastexcel の [GitHub ページ](https://github.com/CodePhiliaX/fastexcel) で `Fork` ボタンをクリックして、プロジェクトをあなたの GitHub アカウントにコピーします。例：`https://github.com/<your-username>/fastexcel`。
2. **リポジトリをクローン**：次のコマンドを実行してフォークしたプロジェクトをローカルにクローンします：
   ```bash
   git clone git@github.com:<your-username>/fastexcel.git
   ```
3. **アップストリームリポジトリを設定**：公式リポジトリを `upstream` として設定して、更新を同期しやすくします：
   ```bash
   git remote add upstream git@github.com:CodePhiliaX/fastexcel.git
   git remote set-url --push upstream no-pushing
   ```
   `git remote -v` を実行して設定が正しいか確認できます。

### ブランチの定義

fastexcel では、すべての貢献は `main` 開発ブランチを基にする必要があります。さらに、次のブランチタイプがあります：
- **リリースブランチ**：バージョンリリースに使用されます（例：`0.6.0`, `0.6.1`）。
- **機能ブランチ**：大規模な機能の開発に使用されます。
- **ホットフィックスブランチ**：重要なバグ修正に使用されます。

PR を提出する際は、変更が `main` ブランチに基づいていることを確認してください。

### コミットルール

#### コミットメッセージ
コミットメッセージが明確で記述されていることを確認し、次の形式に従ってください：
- **docs**: ドキュメントの更新、例：`docs: PR 提出ガイドの更新`。
- **feature**: 新機能、例：`feature: 並行書き込みのサポート`。
- **bugfix**: バグ修正、例：`bugfix: Null ポインタ例外の修正`。
- **refactor**: コードのリファクタリング、例：`refactor: データ処理ロジックの最適化`。
- **test**: テストの追加または改善、例：`test: ユニットテストの追加`。

次のような曖昧なコミットメッセージはお勧めしません：  
- ~~問題を修正~~  
- ~~コードを更新~~  

ヘルプが必要な場合は、[Git コミットメッセージの書き方](http://chris.beams.io/posts/git-commit/) を参照してください。

#### コミット内容
1回のコミットには完全かつレビュー可能な変更が含まれていることを確認してください：
- 大規模な変更を避ける。
- 各コミットが独立しており、CI テストで合格することを確認してください。

また、コミット時に正しい Git ユーザー情報が設定されていることを確認してください：
```bash
git config --get user.name
git config --get user.email
```

### PR の説明

レビュワーが PR の内容と目的を迅速に理解できるように、[PR テンプレート](./.github/PULL_REQUEST_TEMPLATE.md) を使用してください。詳細な説明はコードレビューの効率を大幅に向上させます。

## テストケースの貢献

すべてのテストケースの貢献は歓迎されますが、特にユニットテストが重要です。対応するモジュールの `test` ディレクトリに `XXXTest.java` ファイルを作成することをお勧めします。JUnit5 フレームワークの使用を推奨します。
